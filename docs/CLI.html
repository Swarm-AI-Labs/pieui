Архитектура и диспетчер

Оба CLI — один процесс с плоским switch по верхнему command-токену. Никаких фоновых сервисов, никакого глобального состояния, кроме .pie/config.json и .env на диске. PIE служит источником истины для surface (имена команд, позиционные аргументы, флаги); PIEUI зеркалит surface через TS-реализацию. Постхук .claude/hooks/cli-symmetry-check.sh подсвечивает дивергенции при правках.

TypeScript

src/cli.ts — точка входа (shebang #!/usr/bin/env node). Поток:

  • Срез process.argv.slice(2) → rawArgs
  • detectHelpScope(rawArgs) — короткое замыкание на help
  • parseArgs(rawArgs) → структура ParsedArgs
  • Большой switch(command) с вложенными if (cardAction === …) и if (pageAction === …)
  • Обёртка void main().catch(…) печатает [pieui] Error: … и выходит с кодом 1
switch (command) {
  case 'init': await initCommand(outDir); return
  case 'card': { /* … */ }
  case 'page': { /* … */ }
  default: printUsage(); process.exit(1)
}
Python

pie/__main__.py — точка входа (python -m pie). Поток:

  • build_parser() создаёт argparse.ArgumentParser с subparsers (required=True)
  • parse_args(argv).vars() → **kwargs в run()
  • run() — каскадные if command == "card" → if card_command == "add" и т. д.
  • Все хендлеры в pie/code/, ловятся ValueError и PieStorageError → print(..., file=sys.stderr); return 1
if command == "card":
    if card_command == "add":
        handle_card_add(...)
    elif card_command == "remote":
        if card_remote_command == "push":
            handle_card_remote_push(...)
Симметрия: surface-уровень pieui <command> ↔ pie <command> идентичен. Реализация — нет (TS делает introspection через TypeScript Compiler API, Py — через модуль ast).

Парсинг аргументов

parseArgs / parse_args

argv[] → ParsedArgs

Превращение строки запуска в типизированную структуру. PIEUI — рукописный парсер на argv.find/findIndex/startsWith; PIE — стандартный argparse.

TypeScript

src/code/args.ts вручную сканирует argv в три прохода:

  1. Глобальные dir-флаги: --out-dir=X | --out-dir X | -o X; то же для --src-dir/-s. Форма с = имеет приоритет над пробельной.
  2. Булевы: --append через argv.includes(...).
  3. Командные хвосты: consumeFlags(tail) — линейный walker, потребляющий --ajax, --io, --user/--project, --page/--per-page/--from/--to, --out и их =-формы; всё остальное — позиционно в positionals[].

Дефолты: outDir = "public" для postbuild, иначе "."; srcDir = "." всегда.

Числовые флаги (--page, --per-page, …) валидируются регуляркой /^-?\d+$/ в parseIntFlag(); невалидный ввод бросает Error.

Тип результата: ParsedArgs в src/code/types.ts — все поля опциональны кроме command/outDir/srcDir/append.

Python

argparse.ArgumentParser(prog="pie") в build_parser() определяет всю иерархию:

  • Subparsers: web, card, page, create, init, login, self-upgrade
  • Вложенные subparsers: card_subparsers (add/list/…/remote) → card_remote_subparsers (push/pull/…/private)
  • Типизация флагов: --page/--per-page/--from/--to с type=int; --io/--ajax — action="store_true"
  • Дефолты задаются через default=… на каждом argument-вызове

Результат — argparse.Namespace, далее vars(ns) распаковывается в run(**kwargs).

Невалидный choice (например, card add wrongtype) argparse сам печатает ошибку и выходит с кодом 2.

⚠ Сторона TS жёстче в позиционных правилах: первый позиционный для card add сначала проверяется на принадлежность VALID_COMPONENT_TYPES, и только если не подошёл — трактуется как имя с типом по умолчанию complex-container. У pie тип всегда обязательный позиционный, выбор валидируется argparse-ом.

Источники конфигурации

Конфиг приходит из трёх слоёв (по приоритету сверху вниз): CLI-флаги → переменные окружения (process.env / os.environ) → .env в cwd → .pie/config.json → встроенные дефолты.

TypeScript

src/code/services/settings.ts → loadSettings():

  • parseDotenv() читает .env построчно (своя реализация, без зависимостей)
  • pick(key) — сначала process.env[key], затем значение из .env
  • Возвращает: userId, apiKey, project (fallback — basename(cwd)), componentsDir (default piecomponents/, абсолютный путь), apiBaseUrl (default https://cdn-pieui.swarm.ing/api)

.pie/config.json читается отдельно — там лежат backendPagesDir, backendComponentsDir (заполняются командой init) и кэш креденшелов от login.

Python

pie/config.py → Settings (@dataclass, кэширован @lru_cache):

  • python-dotenv подгружает .env на импорт модуля
  • Поля читаются из os.environ: PIE_USER_ID, PIE_API_KEY, PIE_PROJECT/PIE_PROJECT_SLUG, PIE_COMPONENTS_DIR (default pages/components)
  • api_base_url зашит константой: https://cdn-pieui.swarm.ing/api

.pie/config.json хранит линковку с фронтенд-проектом (frontendProjectDir, frontendComponentsDir) и креды от login.

Идея .pie/config.json: переживает между запусками, не уходит в git (обычно в .gitignore), хранит интерактивные ответы юзера и токены. Идея .env: пишется командой login на одном уровне с .pie/, чтобы остальные скрипты тоже видели креды.

Help-система

Оба CLI поддерживают --help/-h на каждом уровне команды.

TypeScript

detectHelpScope(argv) в src/code/args.ts:

  • Возвращает 'all' | 'card' | 'card-remote' | 'page' | 'init' | 'postbuild' | 'login' | 'create' | null
  • Определяет scope по первым 1–2 токенам (card remote --help → 'card-remote')

printUsage(scope) рендерит срез строк (CARD_LINES, CARD_REMOTE_LINES, PAGE_LINES, …). Перехват сделан в main() до parseArgs.

Python

Бесплатно от argparse: каждый subparser автоматически принимает -h/--help и печатает usage своего уровня.

Текст хелпа собирается из аргументов help= при объявлении subparser-ов и опций.

login

pie login · pieui login

Авторизация через браузер

Бесконечный поллинг credentials-endpoint, пока пользователь не подтвердит сессию в браузере.

TypeScript

src/code/commands/login.ts:

  1. Генерит 32-символьный код ([a-zA-Z0-9])
  2. Открывает https://pieui.swarm.ing/connect?code=<code> (override: PIEUI_LOGIN_CONNECT_BASE)
  3. Поллит https://api-pieui.swarm.ing/api/external/credentials?code=<code> раз в 3 с, таймаут 30 с (override через env)
  4. Ожидает { status: "ok", config: { user_id, project, api_key } }
  5. Пишет ~/.pie/config.json и аппендит PIE_USER_ID/PIE_PROJECT/PIE_API_KEY в .env с экранированием
Python

pie/code/login.py: тот же протокол, тот же эндпоинт, тот же 3-секундный интервал. Сохраняет креды в .pie/config.json, аппендит .env с теми же ключами.

Маппинг ключей JSON↔ENV: user_id → PIE_USER_ID, project → PIE_PROJECT, api_key → PIE_API_KEY.

create / create-pie-app / create-pieui

Скаффолд нового проекта с нуля (создаёт директорию и наполняет её).

TypeScript

pieui create <Name> создаёт Next.js-приложение и затем запускает внутри pieui init. create-pie-app / алиас create-pieui — тот же шаблон без последующего init (чистый скелет).

Хендлер: src/code/commands/create.ts, createPieApp.ts.

Python

Хендлер: pie/code/create.py.

init

Инициализация структуры PieUI в существующем проекте + интерактивная линковка двух репозиториев (фронт ↔ бэк).

TypeScript

src/code/commands/init.ts делает:

  1. Создаёт piecomponents/ в --out-dir (default — cwd)
  2. Пишет piecomponents/registry.ts с заглушкой "use client"
  3. Патчит tailwind.config.{js,ts}: добавляет ./node_modules/@swarm.ing/pieui/dist/**/* в content[]
  4. Создаёт/патчит next.config.ts: ключи PIE_* в env, transpilePackages: ["@swarm.ing/pieui"]
  5. Если TTY: ищет в os.homedir() до глубины 2 директории с pages/+components/, через readline предлагает варианты, сохраняет в .pie/config.json ключи backendPagesDir, backendComponentsDir
Python

pie/code/init.py делает:

  1. Создаёт pages/, pages/components/
  2. Копирует web.py и pages/main.py из templates
  3. Если TTY: ищет в домашней директории фронтенд-проекты (с piecomponents/ или app/), сохраняет в .pie/config.json: frontendProjectDir, frontendComponentsDir
Идея кросс-линковки: обе команды init прописывают пути к парному проекту в локальный .pie/config.json. Это нужно для будущих UX-фич типа "запушить компонент в pieui-проект из pie".

self-upgrade

Самообновление CLI до последней версии. Автодетект пакетного менеджера; ручной override через --pm.

TypeScript

src/code/commands/selfUpgrade.ts. Порядок автодетекта: bun → pnpm → yarn → npm. Команды:

bun add  -g @swarm.ing/pieui@latest
npm install -g @swarm.ing/pieui@latest
pnpm add -g @swarm.ing/pieui@latest
yarn global add @swarm.ing/pieui@latest

Запускается через spawn с stdio: 'inherit', чтобы юзер видел вывод PM в реальном времени.

Python

pie/code/self_upgrade.py. Порядок автодетекта: uv tool → poetry → pip. Команды:

uv tool upgrade pieui
poetry update pieui
pip install --upgrade pieui

card add

Скаффолд нового компонента из шаблона. 4 базовых типа × опции --io × --ajax.

TypeScript

src/code/commands/add.ts. Имя валидируется регуляркой /^[A-Z][a-zA-Z0-9]+$/. Создаёт:

  • piecomponents/<Name>/index.ts — вызов registerPieComponent({ name, component })
  • piecomponents/<Name>/types/index.ts — интерфейс данных (поля зависят от типа + опций)
  • piecomponents/<Name>/ui/<Name>.tsx — скелет React-компонента

Опции:

  • --io → добавляет useSocketioSupport, useCentrifugeSupport, useMittSupport, centrifugeChannel
  • --ajax → добавляет pathname, depsNames, kwargs

Дополнительно: дописывает строку import "@/piecomponents/<Name>"; в registry.ts.

Python

pie/code/card_add.py. Имя нормализуется в CamelCase (my-card → MyCard), файл — в snake_case_card.py.

Шаблон выбирается по формуле {prefix}{card_type}_component.py.j2, где prefix ∈ {"", "io_", "ajax_", "ajax_io_"} и опционально input_. Итого 32 комбинации в pie/code/templates/pages/components/.

Рендерится Jinja2 с переменной {{name}}. Опции:

  • --io → добавляет use_socketio_support, use_centrifuge_support, use_mitt_support, centrifuge_channel
  • --ajax → добавляет pathname, deps_names, kwargs
Типы компонентов: simple (только data), complex (data+children), simple-container/container (data+content), complex-container (data+content[]). В TS дефолт при пропуске типа — complex-container; в Py тип обязателен.

card list

Перечисление зарегистрированных компонентов с типом, AJAX/IO-флагами и путём к файлу.

TypeScript

src/code/commands/list.ts:

  1. glob ${srcDir}/**/*.{ts,tsx}, исключая .d.ts, dist/, node_modules/
  2. Создаёт ts.Program и TypeChecker
  3. Walk AST: ищет CallExpression с callee.text === "registerPieComponent"
  4. Из аргумента-объекта берёт ключи name, component, loader
  5. Через TypeChecker резолвит type компонента, читает свойство content/children для определения типа
  6. Печатает таблицу: Name · Type · DataType · Lazy · File
Python

pie/code/card_list.py:

  1. Рекурсивный обход {components_dir}/**/*.py
  2. Парсинг каждого файла через ast.parse()
  3. Строит alias-map импортов (import X as Y, from M import A as B) для резолва базовых классов
  4. Находит классы, наследующие Card/InputCard (с учётом алиасов)
  5. Сканирует AnnAssign/Assign для определения AJAX (pathname/deps_names/kwargs) и IO (use_*_support) полей
  6. Тип определяет по аннотациям: list[Card] → container, вложенный Card → complex, иначе simple
⚠ TS использует Compiler API → надёжно резолвит типы через checker. Py использует только ast (статический парсер) → не выполняет код, но требует точных regex/AST-эвристик для базовых классов.

card view

Дамп всех видимых снаружи характеристик карточки: props, ajax, IO, events.

TypeScript

src/code/commands/cardView.ts:

  • Читает piecomponents/<Name>/types/index.ts и index.ts/index.tsx
  • Regex interface ${Name}Data { … } вырезает тело интерфейса
  • Regex ^[ \t]*([A-Za-z_]\w*)(\?)?\s*:\s*([^;\n]+);? — список props
  • AJAX: имя props в {pathname, deps_names, kwargs}
  • IO: \buse_[A-Za-z0-9_]+_support\b
  • Events: из methods = { … } в JSX — ключи объекта через regex ([A-Za-z_]\w*)\s*:
Python

pie/code/card_inspect.py:

  • Парсит файл через ast.parse(), находит ClassDef с нужным именем
  • Props — поля AnnAssign в теле класса; default/required определяется наличием value
  • AJAX/IO — те же имена полей
  • Events: get_supported_events() → парсит Return-литерал; плюс методы create_<event>_event с экстракцией параметров из сигнатуры

card pull

card dump-metadata

Канонический JSON-дамп компонента для интероп-сценариев (LLM, миграции, AI-инспекция). Формат описан отдельно в PieMetadata.

TypeScript

src/code/commands/cardDumpMetadata.ts. Собирает 12 полей:

  • name, files[] (все .ts/.tsx кроме тестов/сториз)
  • packages[] — npm-пакеты с версиями из package.json
  • relativeImports[] — резолв через tsconfig.paths
  • events[] — через extractEvents() из methods={…}
  • propsSchema/propsCode — JSON Schema + источник интерфейса данных
  • eventsPropsSchema/eventsPropsCode — payload-типы каждого события
  • inputPropsCode/inputPropsSchema — тип stored={…} (если есть)
  • ajaxList[] — поля с префиксом pathname

Schema генерится через typescript-json-schema с настройками strictNullChecks, noExtraProps, ref: false.

Опция --out <file> — записать в файл; иначе stdout.

Python

pie/code/card_dump_metadata.py. Те же 12 полей:

  • files[] — .py-файлы компонента
  • packages[] — фильтр sys.stdlib_module_names + проверка локальных модулей; версия через importlib.metadata.version() или pyproject.toml
  • relativeImports[] — резолв относительных и абсолютных проектных импортов в пути файлов
  • events[] — пересечение get_supported_events() и create_*_event методов
  • propsSchema/propsCode — schema через metadata_schema.py (рекурсивный AST-only обход аннотаций), source — целиком @dataclass класс
  • Аналогично — eventsPropsSchema/Code для каждого create_*_event метода

Опция --out аналогично.

Оба дампа — детерминированный JSON (отсортированные ключи) для diff-friendly артефактов и сравнения «фронт ↔ бэк» одного и того же компонента.

card remote · push/pull/list/remove/history/public/private

Подкоманды для синка компонентов с PieUI Storage. Один HTTP-клиент в каждой реализации, единый набор эндпоинтов.

TypeScript

src/code/services/storage.ts — обёртка над нативным fetch() с AbortController (30 с таймаут). Аутентификация: заголовок x-api-key из settings.apiKey. Папка cardRemote/ содержит файл на каждый action.

Push: walk файлов компонента, FormData с object_paths[] и files[], PUT batch-эндпоинт; затем отдельный PUT метаданных eventSchema.

Python

pie/code/services/storage.py — синхронный httpx.request() (30 с timeout), та же схема x-api-key. Все вызовы оборачиваются в PieStorageError (subclass RuntimeError).

Push/pull/list/etc. — методы PieStorageService, дёргаются из card_remote.py.

Источники user_id/project при remote-операциях (порядок приоритета):

  1. Флаги --user / --project
  2. Префикс ссылки (для pull): project/Name или r/user/Name
  3. process.env/os.environ: PIE_USER_ID, PIE_PROJECT или PIE_PROJECT_SLUG
  4. .env в cwd
  5. Дефолт project: basename(cwd)

Подкоманда history принимает --page N (default 1), --per-page N (default 10), --from R, --to R — все передаются как query-параметры эндпоинту истории.

page ajax · add / remove

Добавление/удаление ajax-хендлера в существующую страницу через правку исходника.

TypeScript

src/code/commands/pageAjax.ts: на фронте ajax-хендлеры живут на бэке; команда делегирует через subprocess в pie, чтобы сохранить симметрию surface — флаг add/remove, имя handler-а; редактирование выполняется на бэк-стороне.

Python

pie/code/page_create.py:

  • add — добавляет атрибут {handler}_ajax = "/{handler}", вызов self.register_ajax(...) после super().__init__(), и async-метод async def {handler}(self, **data): return []
  • remove — снимает все три участка по позициям AST (с захватом пустой строки перед методом, если есть)

postbuild · web build

Генерация манифеста компонентов для рантайма / статической раздачи. Главный артефакт пайплайна сборки.

TypeScript

pieui postbuild [--out-dir public] [--src-dir src] [--append] → src/code/commands/postbuild.ts:

  1. Glob ${srcDir}/**/*.{ts,tsx}
  2. Создаёт ts.Program, ходит AST в поисках registerPieComponent({…})
  3. Для каждого: имя из name, data-тип из сигнатуры компонента (плюс heuristic-fallback по именам {Name}Data, {TypeName}Data)
  4. Генерит JSON Schema через typescript-json-schema
  5. Если --append: подмердживает node_modules/@swarm.ing/pieui/dist/pieui.components.json (дедуп по имени)
  6. Пишет ${outDir}/pieui.components.json
Python

pie web <module:app> build → pie/code/build.py (handle_build):

  1. Импортирует module:attribute (FastAPI-приложение)
  2. Итерирует страницы из web.pages
  3. Для каждой вызывает page.fields.generate() — рекурсивная сериализация дерева карточек в JSON
  4. Пишет build.json с ключами по page key

Симметричная команда pie web <m:a> verify просто проверяет, что у всех страниц определён .fields.

Интроспекция AST

Главная общая черта обеих CLI: никогда не исполнять пользовательский код (за исключением pie page view, где это сделано осознанно). Всё извлечение метаданных — статический анализ.

TypeScript

Основной инструмент — TypeScript Compiler API (typescript + typescript-json-schema), обёрнутый в src/code/ts.ts. Подходы:

  • Walk AST: ts.forEachChild(node, visit) с матчингом на CallExpression, JsxElement, InterfaceDeclaration
  • TypeChecker: checker.getSymbolAtLocation, checker.getTypeOfSymbolAtLocation — резолв props компонентов через type-граф
  • Schema generation: TJS.buildGenerator(program, {strictNullChecks, noExtraProps, ref: false, ignoreErrors: true})

Регэксы используются только в простых случаях (cardView.ts — для быстрого, не type-aware дампа). Полная интроспекция (postbuild, dump-metadata) — всегда через checker.

Извлечение событий: extractEvents.ts — рекурсивный поиск <PieCard methods={…}>, извлечение ключей object-literal-а. Запрещены spread и computed properties.

Python

Основной инструмент — стандартный модуль ast. Подходы:

  • Walk AST: ast.walk(tree) или ручной обход с ast.NodeVisitor
  • Alias map: словарь {local_name: dotted_path} из всех ast.Import/ast.ImportFrom — нужен для резолва базовых классов (X(Card) ↔ import Card as X)
  • Schema generation: metadata_schema.py — собственный walker, превращающий аннотации в JSON Schema без выполнения кода (поддержка Optional, List, Dict, PEP 604 union X | Y, форвард-рефы как строки, вложенные dataclass)
  • Position tracking: ast.AST.lineno/end_lineno для точечной правки исходников (page ajax, add-event)

Stdlib-фильтр: sys.stdlib_module_names (Python ≥3.10) с fallback на sysconfig.get_paths().

Симметричный набор полей: AJAX (pathname/deps_names/kwargs snake_case в Py, camelCase pathname/depsNames/kwargs в TS), IO (use_*_support/centrifuge_channel ↔ use*Support/centrifugeChannel), events (имена методов create_*_event ↔ ключи methods={…} в JSX).

HTTP API: PieUI Storage

База URL: https://cdn-pieui.swarm.ing/api. Аутентификация: заголовок x-api-key (опускается для публичных GET-ов). Таймаут: 30 с. Ошибки сворачиваются в PieStorageError (Py) / StorageError (TS) и печатаются как [pieui] Error: … / [pie] Error: ….

Шаблоны

TypeScript

Шаблоны живут в коде как функции, возвращающие строку:

  • src/code/templates/componentIndexTemplate.ts — index.ts карточки
  • src/code/templates/componentTypesTemplate.ts — types/index.ts с интерфейсом, поля зависят от типа + --io/--ajax
  • src/code/templates/componentTemplateFor.ts — ui/<Name>.tsx с React-скелетом под каждый тип
  • src/code/templates/pageTemplate.ts — app/<path>/page.tsx
  • src/code/templates/registryTemplate.ts — piecomponents/registry.ts

Подстановка переменных — обычный ${name} в template literals; никакого Jinja-подобного движка.

Python

Шаблоны — файлы .j2 в pie/code/templates/, рендерятся Jinja2.

Карточки: 4 базовых типа × 8 префиксов = до 32 файлов в templates/pages/components/ (формула: [input_][ajax_][io_]{type}_component.py.j2).

Прочее:

  • templates/page.py.j2 — AsyncPage-подкласс
  • templates/main.py.j2 — pages/main.py
  • templates/web.py.j2 — каркас FastAPI с условными блоками socketio/centrifuge
